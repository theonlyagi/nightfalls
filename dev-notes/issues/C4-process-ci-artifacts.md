# C4 — Process risk: no CI, built artifacts committed, direct-to-main merges

**Severity:** Critical (process, not code)
**Area:** repo hygiene / release process

## What's happening

A full-file rewrite from a stale local checkout silently clobbered an
already-merged PR's server logic (weapon/mutation + earlier day/night server
code) and was pushed **straight to `main`, bypassing PR review**
(merge `a149fdd`, deploy commit `b7c519a`).

Two root causes made this possible and hard to catch:

- **Built artifacts are committed to git.** `public/game.js` and
  `docs/index.html` are generated by the build, but tracked in the repo — so
  they show up in every diff, conflict constantly, and go stale relative to
  source. CLAUDE.md already documents this having bitten the project before.
- **No CI.** Nothing runs a typecheck or build on push/PR, so a regression
  (or a rewrite that drops merged work) reaches `main` — and the live site —
  with no automated gate.

## Why it matters

This is the most impactful *process* fix available. Without it, any dev
working from an out-of-date branch can overwrite merged work, and it only gets
noticed by manually reading the deployed code.

## Suggested direction

- Add a minimal CI check (typecheck + `npm run build` for both client and
  server) that must pass before merge.
- Stop tracking built artifacts: `.gitignore` `public/game.js` and generate
  `docs/index.html` at deploy time instead of committing it. (This interacts
  with the GitHub Pages setup — see CLAUDE.md's notes on `build:share` — so
  it needs a deliberate plan, not just a delete.)
- Protect `main` so changes land via PR rather than direct pushes.

Flagged for team decision; none of this has been changed.
